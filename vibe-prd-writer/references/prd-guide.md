# PRD Writing Guide

## Effective PRD Best Practices

### 1. Start with User Needs
- Always begin with "Who are we building this for?"
- Focus on user problems, not solutions
- Use specific user personas

### 2. Define Clear Problems
- Be specific about current pain points
- Quantify the impact if possible
- Set measurable success metrics

### 3. Keep it Concise
- Aim for one page maximum
- Use bullet points and clear sections
- Avoid technical jargon when possible

### 4. Set Boundaries
- Clearly state what's out of scope
- Define constraints upfront
- Prevent scope creep

### 5. Technical Considerations
- Choose appropriate technology stack
- Consider dependencies and constraints
- Plan for scalability

## Common Pitfalls to Avoid

- ✗ Don't mix requirements with solutions
- ✗ Don't skip user research
- ✗ Don't make it too vague
- ✗ Don't forget success metrics
- ✗ Don't ignore technical constraints

## Example Structure

```
# Product Requirements Document: [Name]

## 1. Target Users
- Primary: [Persona]
- Secondary: [Groups]
- Needs: [Specific needs]

## 2. Problem Statement
- Current Pain: [Specific problems]
- Impact: [Business value]
- Success Metrics: [KPIs]

## 3. Core Features
- Feature 1: [Description]
- Feature 2: [Description]
- Feature 3: [Description]

## 4. Scope Boundaries
- Out of Scope: [Exclusions]
- Constraints: [Limitations]
- Timeline: [Timeline]

## 5. Technical Approach
- Technology: [Stack]
- Dependencies: [Key deps]
- Approach: [Strategy]
```

This structure ensures your PRD is:
- User-focused
- Problem-driven
- Actionable
- Measurable
- Implementable
